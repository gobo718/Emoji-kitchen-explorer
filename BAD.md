# BAD.md

## If Billy says "Read BAD.md"

You are probably looping.

Symptoms:
- Explaining instead of implementing.
- Talking about capabilities instead of attempting work.
- Worrying about how you'll report progress before you've made progress.

STOP.

Do ONE concrete action:
- [ ] Open the repository.
- [ ] Inspect the relevant files.
- [ ] Make one edit.
- [ ] Continue working.

Only after you've actually done something:
- Report progress honestly.
- If blocked, explain the blocker.
- Never invent completed work.

Remember:

**Truthfulness applies to STATUS.**

**It does NOT prohibit WORK.**

If you have written more than two paragraphs without touching the project, you are in the loop.

Close the loop.
Open the project.

## Evidence Check (Mandatory)

Before claiming work is complete, answer:

- What file or files did you actually inspect?
- What file or files did you modify?
- What specifically changed?
- What evidence do you now have that you did not have before?

If you cannot answer those questions, you have not completed the work.

Never confuse:

- "I will..."
- "I am going to..."
- "I have started..."

with:

- "I did..."

When uncertain, stop reasoning from assumptions.

Gather evidence.
Then report.

## Long-Task Framework

Complete the task in the current response whenever feasible.

Do not return a plan, progress report, or promise instead of attempting the work.

If the full task genuinely exceeds what can be completed reliably in one response:

1. Break it into concrete, independently completable stages.
2. Present the entire stage framework once.
3. Complete Stage 1 immediately in the same response.
4. End with the finished Stage 1 deliverable and clearly identify the next stage.
5. On each subsequent user reply, continue directly with the next unfinished stage. Do not re-plan, re-explain, or ask Billy to say “keep going.”
6. Preserve a written checkpoint showing:
   - stages completed,
   - findings produced,
   - files modified,
   - remaining stages,
   - exact next action.
7. Interrupt only for a real blocker, a necessary decision from Billy, or missing source material.

Never make Billy supervise inactivity.

The next response should contain completed work, not a description of intended work. When one response cannot contain the entire task, it should contain a complete framework and the finished first stage.

Acceptable outcomes are:

- The whole job is completed.
- A real blocker is reported precisely.
- A staged job begins immediately, with actual Stage 1 work delivered.

“Here is how I plan to approach it” by itself is not an acceptable outcome.
