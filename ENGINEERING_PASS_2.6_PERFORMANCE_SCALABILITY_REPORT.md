# Engineering Pass 2.6 — Performance & Scalability Report

## Scope
Billy Labs v2.5.5 was audited at the current 147,000-row Emoji Kitchen inventory size. The pass covered canonical identity construction, exact resolution indexing, ingredient lookup, text search, random discovery, and newest-release discovery. UI behavior, ranking, eligibility, URLs, stored IDs, and inventory data were held constant.

## Findings
The engine was structurally sound but still repeated expensive work in three places:

1. Canonical identity repeatedly reconstructed Unicode code-point comparison keys.
2. Ingredient pages scanned the full inventory every time a base emoji page opened.
3. Search and discovery rebuilt overlapping normalized or filtered data during repeated interactions.

## Changes
- Canonical code-point keys are cached by normalized emoji.
- Canonical IDs are cached by inventory row through `BillyMashups.rowId(row)`.
- Resolution index construction consumes cached row IDs.
- Ingredient-to-row indexes are cached per inventory array.
- Search text indexes are cached per inventory array and stable text accessor.
- Discovery eligibility is evaluated in one pass rather than separate exclusion and unseen passes.
- Newest-date analysis and its matching row group are cached per inventory array.
- Public methods still return fresh arrays where callers previously received independent arrays.

## Benchmark snapshot
Generated by `tools/performance-benchmark.js` against all 147,000 rows. Measurements vary by machine and are diagnostic rather than hard release thresholds.

- Resolution index cold build: approximately 103 ms median.
- 1,000 cached exact resolutions: approximately 1.7 ms median.
- Ingredient index cold build: approximately 39 ms median.
- Cached ingredient lookup: approximately 0.002 ms median.
- Repeated full-inventory text search after index creation: approximately 5 ms median.
- Cached newest-release group lookup: approximately 0.001 ms median.
- Unseen-preferred discovery: approximately 200 ms median because current seen state must be evaluated across the eligible inventory.

## Scalability boundary
The current architecture is appropriate for a static in-browser inventory of 147,000 rows. Search indexing trades additional browser memory for responsive repeated searches. Progress-sensitive discovery intentionally remains a live scan so recently viewed mashups are respected immediately.

If the inventory grows substantially beyond the current scale, the next justified step would be generated build-time indexes or a Worker-backed query layer, not additional page-local caches.

## Compatibility
No mashup data, canonical IDs, URLs, discovery policies, search fields, sort behavior, favorites, seen state, blurblets, or cloud contracts were changed.
